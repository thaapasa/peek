# Architectural Review — 2026-04-26

Snapshot review of the codebase at commit `806ab38` (main). Code compiles
cleanly, all 45 tests pass, no clippy warnings. Findings below are about
deeper structural/correctness issues.

## Overall

Architecture is clean and the recent Mode-stack refactor is good — `Mode`
trait + `ViewerState` + a single event loop is the right shape, and
`compose_modes` does most of the per-file-type dispatch in one place.
Position propagation across mode switches via `Position::{Byte, Line}` is
a nice touch. The `Viewer`/`Mode` split (piped vs. interactive) is a
defensible separation.

---

## Architectural concerns

### 1. The "stream, don't load" rule is broken in three load-bearing places

For a viewer that markets itself for multi-GB files, several hot paths
read the entire file:

- **`input/detect.rs:73`** — *FIXED.* `let buf = fs::read(path)?` slurped
  the entire file just to run `infer::get` on the first few bytes and a
  `String::from_utf8` UTF-8 check. Now reads a 16 KB head for `infer`
  and uses a chunked `is_utf8_streaming` that scans the rest in 64 KB
  pieces, carrying incomplete trailing UTF-8 sequences across chunk
  boundaries.

- **`input/source.rs:67 line_to_byte` / `:84 byte_to_line`** — *FIXED.*
  Both called `self.read_bytes()` which loaded the whole file on every
  position-tracking mode switch (Content↔Hex). Now scan via
  `open_byte_source().read_range(...)` in 64 KB chunks. For files this
  is true `seek`+`read` streaming; for stdin it's still a slice copy per
  chunk, but bounded to 64 KB instead of cloning the whole buffer.

- **`viewer/mod.rs:205 text_content_mode`** — `source.read_text()?` is a
  whole-file read. Harder to fix because syntax highlighting wants
  context, but at minimum it should be deferred until ContentMode is
  actually rendered (currently happens inside `compose_modes`, which runs
  even if the user immediately presses `x` and never views the Content
  mode). The pretty-print arms additionally fully re-allocate. Worth
  lazy-loading.

### 2. `main.rs` still has a residual file-type branch for animation

`main.rs:65-86` does
`if matches!(file_type, FileType::Image) && ... decode_anim_frames(...)`.
CLAUDE.md flags this exact pattern as the kind of dispatch
`compose_modes` exists to eliminate. With `Mode::next_tick`/`tick`
already in the trait, animation can compose like everything else:
`compose_modes` decodes once, returns either `[ImageRenderMode, ...]` or
`[AnimationMode, ...]`, and `view_animated` (`image/animate.rs:148`)
becomes redundant with `compose_modes` + `interactive::run`. That also
drops the duplicated help-action union code in `view_animated:160-168`.

### 3. `Position` translation has a semantic hole for pretty-printed content

`source.line_to_byte(line)` counts source-file newlines. But ContentMode
displays *pretty-printed* JSON/YAML/TOML/XML — the line index in the
pretty view does not correspond to source bytes. Switching
ContentMode→HexMode after scrolling pretty JSON jumps to a completely
unrelated byte offset. The doc on `line_to_byte:60-62` admits
"approximate in that case", which understates it.

The clean fix is for ContentMode to expose a real position (e.g.
`Position::Line(source_line)` translated through a pretty→source map, or
simply not track position when `use_pretty` is on).

### 4. `ModeId::ImageRender` is reused by two structs

Both `ImageRenderMode` and `AnimationMode` use `ModeId::ImageRender`
(`modes/image_render.rs:50`, `modes/animation.rs:45`). `mode_index(id)`
returns the first match; the design is order-dependent. Today there's no
collision because only one of them is ever in the stack, but a future
"PDF page + thumbnail" stack would silently break. Either give each a
distinct ID, or document that ImageRender is a "category" and keep them
mutually exclusive in the type system.

### 5. Two parallel composers

`compose_modes` (`viewer/mod.rs:142`) and `view_animated`
(`image/animate.rs:148`) both build a mode stack and both build the same
help-action union. The composition logic — append Hex/Info/Help and union
their extras — should live in one helper. (Folds naturally into fix #2.)

### 6. `dispatch_key` lets modes shadow globals

`ui/state.rs:138` runs extras first, then globals. The comment says "no
conflicts today" — true, but nothing enforces it. If a future mode added
`Quit` to its extras, the user couldn't quit. Either declare globals
always-win, or move global keys (Quit, scroll, theme cycle) out of
extras so a mode can't accidentally shadow them. Cheapest fix: have the
dispatcher dispatch globals first, since none of the mode extras
intentionally shadow globals.

### 7. `InputSource` is `Clone` over `Vec<u8>`

For stdin, `InputSource::Stdin { data: Vec<u8> }` is cloned per mode
(Content + Hex + Info + Animation = 4 copies of the bytes), per
`decode_anim_frames` call, per `open_byte_source` call (`source.rs:95`),
per stdin EXIF/HDR scan, etc. For a 200 MB animated GIF on stdin this is
real memory. Make it `Arc<[u8]>`.

## Bugs

### Correctness

- **`viewer/modes/animation.rs:79,104`** — `(self.current + n - 1) % n`
  and `(self.current + 1) % self.frames.len()` panic if
  `frames.is_empty()`. The constructor doesn't enforce non-empty; only
  `decode_anim_frames` filters `len <= 1 → None`. A defensive guard or an
  assertion in `AnimationMode::new` would close the gap.

- **`viewer/ui/mod.rs:89 strip_ansi_width` / `:107 truncate_ansi`** —
  count `c.chars()` as 1 column each. CJK and emoji are 2 cols;
  combining marks are 0. Status-line composition (filename in the
  segments, hints on the right) misaligns or wraps badly for any file
  whose name contains a wide character. Pull in `unicode-width` or
  accept the limitation explicitly in the doc.

- **`viewer/structured.rs:108 pretty_xml`** —
  `reader.config_mut().trim_text(true)` discards all whitespace text
  nodes. For `.html`/`.xhtml` (which `detect.rs:62` routes to
  `StructuredFormat::Xml`) this is destructive — `<pre>` content
  collapses, inline whitespace between tags is lost. The user can press
  `r` to see raw, but the default view is wrong. Either preserve text or
  warn.

- **`viewer/mod.rs:209-210 text_content_mode`** — `pretty_print` returns
  `Result<String>`. If parsing fails (malformed JSON in a `.json` file),
  `text_content_mode` propagates the error and the *entire* viewer fails
  to start. A bad JSON file becomes unviewable even in raw mode. Should
  fall back to raw with a warning.

- **`input/stdin.rs:24`** — `if args.files.is_empty() && !implicit_stdin`
  simplifies to `args.files.is_empty() && stdin_is_tty`. Works, but the
  indirection through `implicit_stdin` makes the check confusing.
  Inline it.

- **`input/stdin.rs:83 reopen_stdin_from_tty`** —
  `tty_path.as_deref().or(Some("/dev/tty"))` keeps a fallback to
  `/dev/tty` even though the doc above explicitly says it doesn't work
  on macOS with kqueue. The fallback either (a) succeeds on Linux where
  it would have succeeded anyway, or (b) returns a fd that
  crossterm/mio later rejects. Either drop the fallback (and let
  `ttyname` failure mean "no TTY") or document why both branches are
  kept.

- **`viewer/mod.rs:288 fallback_syntax_token`** — the
  `"dockerfile" | "Dockerfile"` arm is dead code in the extension-keyed
  path. The caller in `compose_modes` paths only hits this with
  extension strings, but a Dockerfile usually has *no* extension; the
  file-name-based fallback in `syntax_token_for:265-269` does pass the
  bare name "Dockerfile", but it's case-sensitive and doesn't go through
  `fallback_syntax_token`. Net result: Dockerfiles never get the
  Dockerfile syntax via this fallback. Either route file-name-based
  lookups through this map, or drop the arm.

- **`output/pager.rs:54-58 Output::write_str` for `Buffer`** — appends
  raw text to the last "line", embedding `\n` mid-line. If `Buffer` is
  ever wired up the breakage is invisible. Currently `Buffer` is
  `#[allow(dead_code)]` and unused — recommendation is to delete it
  (per "no half-finished implementations" in collaboration).

### Performance / wastefulness

- **`info/gather.rs:159 gather_image_extras`** — `image::open(path)`
  decodes the entire image just to read `color_type` and `bit_depth`.
  For a 100 MP RAW that's painful. `image::ImageReader::open(path)?` and
  the format introspection methods can give this without a full decode
  in many cases.

- **`info/gather.rs:170 anim_frame_count`** — iterates
  `decoder.into_frames().count()`, which still decodes every frame
  (`image::AnimationDecoder::into_frames` is the same iterator the
  actual viewer uses). This runs every time the Info view renders for an
  animated image. Prefer a header-only count (e.g.,
  `gif::DecodeOptions::new().read_info(...)` for the GIF case).

- **`info/gather.rs:109,170 anim_frame_count` for stdin** —
  `data.to_vec()` clones the buffer just to wrap it in an
  `InputSource::Stdin`. Tied to fix #7 (Arc the buffer).

- **`viewer/image/render.rs:163 add_margin`** — `enumerate_pixels` +
  `put_pixel` per pixel. `image::imageops::overlay` does the same thing
  in a single contiguous copy. Not a bottleneck, but free win.

### Smaller nits

- **`viewer/modes/image_render.rs:22`** — `background` shadows
  `config.background`. After `Action::CycleBackground`,
  `self.background` and `self.config.background` diverge. Render uses
  `self.background` so it works; the duplicate field is a code smell.
  Mutate `config.background` in place, like `AnimationMode` does
  (`modes/animation.rs:84`).

- **`viewer/ui/state.rs:227 apply` catchall `_ => Outcome::Unhandled`** —
  adding a new `Action` variant won't get a compile-time match check
  here. Drop the catchall; have the dispatcher only return Actions that
  are in `GLOBAL_ACTIONS` so this match is exhaustive.

- **`info/render.rs:392 format_time`** — UTC, no timezone label, no
  indication. "Modified 2026-04-26 09:44:00" reading as UTC may surprise
  users in non-UTC zones. Either label as UTC explicitly or convert to
  local (chrono adds weight; a manual offset via `libc::localtime_r` is
  feasible).

- **`viewer/image/glyph_atlas.rs:46 best_glyph` ties** — when `space` is
  processed, an all-zeros cell scores `dist_inverted = 0`, which then
  equals `best_dist`. Subsequent glyphs with `dist_normal == 0` (full
  block on all-ones) replace it correctly because of `<`. Fine, but the
  inverted-space case (` ` with fg/bg swap = visually `bg` block) is
  sometimes selected when full block would match identically — wastes
  ANSI bytes. Order-dependent and harmless visually.

- **`viewer/structured.rs:32-35`** —
  `let format = match file_type { FileType::Structured(f) => *f, _ => return Ok(()) }`.
  Silently no-op on misdispatch. Better: `unreachable!()` since the
  registry only routes Structured here.
