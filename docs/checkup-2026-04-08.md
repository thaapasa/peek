# Architectural Review - Peek Codebase

**Date:** 2026-04-08

## HIGH SEVERITY

### 1. ~~Architectural Violation: Raw ANSI Escapes Outside UI Composition Layer~~ FIXED

Multiple modules directly construct ANSI escape sequences instead of using `PeekTheme::paint()` as
specified in conventions.md:

- **src/viewer/interactive.rs:155** — `render_status_line()` builds raw `\x1b[38;2...` instead of
  using `PeekTheme::paint()`
- **src/viewer/image/animate.rs:311** — `render_anim_status_line()` duplicates the same inline ANSI
  formatting
- **src/viewer/image/animate.rs:350** — Background color escape `\x1b[48;2...` hardcoded
- **src/viewer/image/render.rs:112, 117, 153, 155** — Raw ANSI in `render_block_color()` and
  `render_density()`
- **src/viewer/mod.rs:51** — Raw reset escape `\x1b[0m` in `highlight_lines()`
- **src/viewer/structured.rs:65** — Raw reset escape in structured viewer
- **src/theme.rs:167** — While `PeekTheme::paint()` correctly centralizes foreground colors, these
  other locations bypass it

**Impact:** Future color mode support (256-color, 16-color, monochrome) cannot be applied globally.
The convention states "All colored terminal output must go through `PeekTheme::paint()`."

**Fix:** Extend `PeekTheme` with methods for background coloring and ANSI reset, or create a
dedicated ANSI formatting module. Update all raw escapes to use centralized methods.

## MEDIUM SEVERITY

### 2. ~~Duplication: Status Line Rendering~~ FIXED

Three nearly-identical status line rendering functions with raw ANSI formatting:

- **src/viewer/interactive.rs:151-182** — `render_status_line()`
- **src/viewer/image/animate.rs:301-354** — `render_anim_status_line()`
- Both duplicate the `fg()` helper, ANSI background composition, and status layout logic

**Impact:** Code duplication makes maintenance harder and increases bug surface area.

**Fix:** Extract a shared `compose_themed_status_line()` function in `ui.rs` that accepts left/right
content and theme, handles ANSI composition centrally.

---

### 3. Dead Code: Background::label() Method

**src/viewer/image/mod.rs:51-59** — The `Background::label()` method is marked `#[allow(dead_code)]`
and is never called anywhere in the codebase.

**Impact:** Dead code increases cognitive overhead.

**Fix:** Remove the method or implement the planned feature that uses it (status line background
display).

---

### 4. ViewerState Has Excessive Public Fields

**src/viewer/ui.rs:92-98** — Seven public fields:

- `pub view_mode`
- `pub current_theme`
- `pub peek_theme`
- `pub content_lines`
- `pub info_lines`
- `pub help_lines`
- `pub scroll`

These are accessed directly by `interactive.rs`, `image/animate.rs` for reading, and external code
is modifying them. This violates encapsulation.

**Impact:** Changes to ViewerState internals require updates across multiple modules. No central
place to add invariant checks or logging.

**Fix:** Make these fields private (`pub(crate)` for necessary internal access), expose only getter
methods. Provide builder methods for state transitions.

## LOW SEVERITY

### 5. Inconsistent Error Fallbacks

Multiple uses of `unwrap_or()`, `unwrap_or_default()` for graceful degradation:

- **src/help.rs:24, 44** — `LOGO.iter().map(...).max().unwrap_or(0)` (safe, logo always present)
- **src/info.rs:59, 94** — Extension/filename handling with `.unwrap_or_default()` (safe)
- **src/viewer/ui.rs:273, 387** — Terminal size detection with `.unwrap_or((80, 24))` (safe default)
- **src/viewer/interactive.rs:96, 175** — Filename and terminal width with `.unwrap_or()` (safe
  defaults)

**Status:** These are all defensive and safe. No action needed — they're used correctly in fallback
contexts.

---

### 6. Feature Status Mismatch in Documentation

**docs/features.md** states image mode cycling should be implemented (line 136-151):
> "Four ASCII-art rendering modes (cyclable with `m` in viewer mode)"

However, `m` key handling for ImageMode cycling is **not implemented** in the interactive viewers.
Only CLI flag `--image-mode` works.

**Impact:** User expectations don't match implementation.

**Fix:** Either implement the feature or update docs/features.md to mark it as "Planned" and explain
current status.

---

### 7. Unused Import Warnings

Two unnecessary casts in the example (not in main binary, but worth noting):

- **examples/gen_glyphs.rs:193-194** — Unnecessary `as i32` casts

**Status:** Clippy reports this; fix is trivial but not blocking.

## Clean Findings

**Positive observations:**

- Image rendering pipeline correctly follows resize-before-composite order (architecture.md "Image
  rendering pipeline")
- Event loop architecture is clean: `ViewerState::handle_key()` properly separates shared vs.
  viewer-specific bindings
- File type dispatch via `Registry` pattern works well
- Theme management is centralized via `ThemeManager`
- No `unwrap()` in production code outside safe fallback contexts
- Consistent use of `anyhow::Result` for error propagation
- No raw I/O operations without proper context wrapping

---

## Summary

The codebase is generally well-structured with one significant architectural violation: raw ANSI
escape sequences are scattered across multiple modules instead of being centralized through
`PeekTheme::paint()`. This violates the stated convention and will make future color mode support
difficult. The duplication of status line rendering is a secondary concern that could be cleaned up
during the same refactor.

Recommend fixing issues #1 and #2 before the next release. Issues #3-7 are lower priority and can be
addressed incrementally.
